# Пользовательские сценарии (User Scenarios)

В системе предусмотрены две основные роли пользователей:

- **Читатель (Reader)** — потребляет контент: просматривает, читает, ищет статьи.
- **Публицист (Publisher)** — расширенная роль: читает + создаёт и публикует статьи.

---

# 1. Сценарии роли «Читатель»

## 1.1 Регистрация и вход

**Цель:** получить доступ к системе как авторизованный пользователь.

**Шаги:**

1. Пользователь открывает страницу регистрации.
2. Вводит:
   - имя пользователя
   - логин
   - пароль
   - выбирает роль «читатель»
3. Отправляет форму.
4. Сервер создаёт пользователя в MongoDB.
5. Пользователь автоматически авторизуется (JWT + cookie).
6. Перенаправляется в систему.

---

## 1.2 Просмотр списка статей (Explore)

**Цель:** найти интересные публикации.

**Шаги:**

1. Пользователь открывает страницу Explore.
2. Клиент отправляет запрос на API `/posts/explore`.
3. Сервер возвращает список постов (с пагинацией).
4. UI отображает карточки статей.

---

## 1.3 Просмотр статьи

**Цель:** прочитать выбранную статью.

**Шаги:**

1. Пользователь выбирает пост.
2. Переходит на страницу `/read/:id`.
3. Клиент делает запрос `/posts/read/:id`.
4. Сервер возвращает:
   - текст статьи
   - автора
   - дату
   - медиа (image/pdf)
5. UI отображает статью.

---

## 1.4 Работа с сессией

**Цель:** поддерживать авторизацию.

**Шаги:**

- JWT хранится в cookie.
- При каждом запросе middleware `verifySession` проверяет токен.
- При ошибке 401 пользователь разлогинивается автоматически.

---

# 2. Сценарии роли «Публицист»

## 2.1 Создание публикации

**Цель:** создать новую научную статью.

**Шаги:**

1. Пользователь открывает страницу Publish.
2. Заполняет форму:
   - заголовок
   - описание
   - загружает изображение (multipart/form-data через multer)
   - загружает PDF файл
3. Отправляет запрос `POST /posts/publish`.
4. Сервер:
   - валидирует данные через Zod
   - проверяет JWT
   - создаёт запись в MongoDB
   - сохраняет файлы в `file_storage/posts`
5. Возвращает ID поста.
6. Клиент отображает уведомление через ModalContext.

---

## 2.2 Просмотр своих публикаций

**Цель:** управлять своими постами.

**Шаги:**

1. Пользователь открывает профиль или раздел "My posts".
2. Запрос: `/posts?author=currentUser`.
3. Сервер возвращает список постов пользователя.
4. UI отображает список с возможностью просмотра.

---

## 2.3 Удаление публикации

**Цель:** удалить ранее созданный пост.

**Шаги:**

1. Пользователь нажимает "Удалить".
2. Отправляется `DELETE /posts/:id`.
3. Сервер:
   - проверяет владельца поста
   - удаляет запись из MongoDB
   - удаляет файлы из `file_storage`
4. UI обновляет список.

---

# 3. Общие сценарии системы

## 3.1 Авторизация (JWT + Cookie)

- Сервер выдаёт JWT после login/register
- Токен сохраняется в cookie
- Middleware проверяет валидность токена
- При 401 вызывается logout

---

## 3.2 Обработка ошибок

- 400 — валидация (Zod)
- 401 — не авторизован (JWT)
- 404 — ресурс не найден
- 500 — внутренняя ошибка сервера

---

## 3.3 Загрузка файлов

- multer обрабатывает multipart/form-data
- файлы сохраняются в `file_storage/posts`
- имя файла связано с postId
- доступ через `/static` или `/file_storage`

---

## 3.4 Пагинация

- Используется в `/posts/explore`
- Параметры:
  - `page`
  - `limit`
- Сервер возвращает:

```json
{
  "items": [],
  "page": 1,
  "totalPages": 10
}
```

---

## 3.5 WebSocket (Socket.io)

**(планируемый функционал)**

- уведомления о новых публикациях
- live-обновления ленты
- уведомления о действиях пользователя

---

# 4. Сценарии безопасности

## 4.1 Проверка роли

- Publisher может создавать посты
- Reader не имеет доступа к publish endpoint

## 4.2 Защита маршрутов

- middleware `verifySession`
- проверка JWT
- проверка роли

---

# 5. Сценарий загрузки файлов (multer)

1. клиент отправляет multipart/form-data
2. multer принимает:
   - image
   - pdf

3. сервер:
   - сохраняет временно
   - переименовывает в postId
   - переносит в `file_storage/posts`

4. запись в MongoDB связывает пост и файлы

---

# 6. Итог

Система построена как SPA + REST API + MongoDB, с:

- JWT авторизацией
- файловым хранилищем
- ролями пользователей
- модульной архитектурой (FSD-подход)
- расширяемостью под Socket.io и новые сервисы
