# SWOT-анализ

> Проект: ScienceJournal — Платформа публикации научных статей  
> Группа: ПИЖ-б-о-24-1 | СКФУ, 2026

---

## Матрица SWOT

| | **Положительные факторы** | **Отрицательные факторы** |
|---|---|---|
| **Внутренние** | **Сильные стороны (S)** | **Слабые стороны (W)** |
| **Внешние** | **Возможности (O)** | **Угрозы (T)** |

---

## S — Сильные стороны (Strengths)

| # | Фактор | Описание |
|---|---|---|
| S1 | **Современный технологический стек** | React + Node.js + MongoDB — распространённый MERN-стек, хорошо документированный и востребованный в индустрии |
| S2 | **JWT-аутентификация** | Токены хранятся в HTTP-only cookies (защита от XSS), реализована полная цепочка: register → auth → verify → logout |
| S3 | **FSD-архитектура фронтенда** | Feature-Sliced Design обеспечивает чёткое разделение UI / бизнес-логики / API-слоя |
| S4 | **Мультизагрузка файлов** | Поддержка одновременной загрузки изображения и PDF через Multer (multipart/form-data) |
| S5 | **Ролевая модель** | Два типа пользователей (publisher / reader) с различными правами доступа |
| S6 | **Валидация данных** | Zod-схемы на бэкенде предотвращают некорректные данные на входе в API |
| S7 | **Пагинация** | Лента статей поддерживает постраничную навигацию (page + limit) |
| S8 | **Изоляция слоёв** | Независимые `client/` и `server/` — фронтенд и бэкенд могут развёртываться раздельно |

---

## W — Слабые стороны (Weaknesses)

| # | Фактор | Описание |
|---|---|---|
| W1 | **Отсутствие поиска** | Нет полнотекстового поиска по названию/содержимому публикаций |
| W2 | **Нет административной панели** | Модерация контента возможна только через прямое обращение к MongoDB |
| W3 | **Локальное файловое хранилище** | Файлы хранятся на диске сервера — при масштабировании нужен CDN |
| W4 | **Отсутствие refresh-токена** | JWT имеет фиксированный срок действия без механизма автообновления |
| W5 | **MongoDB без транзакций** | При сбое при публикации файл может быть сохранён, а запись в БД — нет |
| W6 | **Нет автоматических тестов** | Отсутствует тест-набор (unit/integration tests) для backend и frontend |

---

## O — Возможности (Opportunities)

| # | Фактор | Описание |
|---|---|---|
| O1 | **Socket.IO real-time** | Уведомления о новых публикациях в реальном времени (планируется) |
| O2 | **Redis кэширование** | Кэш для `GET /api/posts/explore` снизит нагрузку на MongoDB при росте аудитории |
| O3 | **CDN / S3 для файлов** | Перенос медиафайлов в объектное хранилище (AWS S3, MinIO) |
| O4 | **Полнотекстовый поиск** | MongoDB Atlas Search / Elasticsearch для поиска по публикациям |
| O5 | **PWA + offline-режим** | Service Worker для кэширования прочитанных статей |
| O6 | **CI/CD пайплайн** | Автоматическое тестирование и деплой через GitHub Actions |
| O7 | **Комментарии и рейтинги** | Расширение предметной модели для социального взаимодействия |

---

## T — Угрозы (Threats)

| # | Фактор | Описание |
|---|---|---|
| T1 | **Безопасность файлов** | Multer без проверки MIME-типа может принять вредоносный файл под видом PDF |
| T2 | **JWT без blacklist** | Отозванный токен остаётся валидным до истечения срока — нужен Redis blacklist |
| T3 | **Рост объёма MongoDB** | Без индексов на `author` и `publishDate` запросы замедлятся при большом объёме |
| T4 | **CORS в production** | Хардкод `http://localhost:5173` требует изменения при деплое на реальный домен |
| T5 | **Конкуренция** | ResearchGate, Academia.edu предоставляют аналогичный функционал на промышленном уровне |

---

## Стратегии на основе SWOT

| Стратегия | Описание |
|---|---|
| **SO**: S2 + O1 | JWT + Socket.IO = защищённые real-time уведомления |
| **WO**: W3 + O3 | Заменить локальный fs на S3-совместимое хранилище |
| **ST**: S6 + T1 | Расширить Zod-валидацию для проверки MIME-типов загружаемых файлов |
| **WT**: W4 + T2 | Реализовать refresh-token механизм с Redis blacklist |
